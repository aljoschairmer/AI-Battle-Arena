## Cross-cutting findings (span multiple files)

1. **Priority-1 survival is the least threat-aware code in the engine, and it preempts dodge.** `survivalBehavior`'s zone-return (survival.ts #1/#2) outranks `emergencyDodge` yet is the only major movement decision with *zero* threat-field consultation of any kind ŌĆö not even the narrower per-tile checks `combat.ts` #6 uses.
2. **Two independent, non-communicating "back off" thresholds exist**: `retreatAndHeal`'s static `hpRetreatFraction` (survival #6) and controller.ts's trade-aware `shouldEngage` gate at hp<0.6 (only reachable while a target is selected and unforced). Neither informs the other; a full-health bot in a confirmed-bad 2v1 never triggers either one.
3. **Grapple/shove are 100% offensive today** (combat #7), contradicting the file's own docstring, and there is no fallback anywhere in `survival.ts` for a cornered, confirmed-losing bot to use them for separation.
4. **Weapon-matchup knowledge (`matchups.ts`) and defense-aware damage math (`derived.ts`) are both fully implemented but architecturally disconnected from `targeting.ts`/`combatMath.ts`** ŌĆö the data exists, the wiring doesn't (Phase 0 ┬¦3/┬¦4, targeting #1).
5. **Dagger backstab is dead-code-gated** (combat #2) ŌĆö the single clearest, highest-confidence logic bug found in Phase 1.

Everything marked CONFIRMED above is a static-analysis conclusion from control flow and data flow, not yet a measured win-rate cost ŌĆö that's what Phase 2 telemetry is for, particularly the two INCONCLUSIVE/PARTIAL items (charged-shot interrupt risk, dodge candidate-set narrowness) which can't be resolved by reading code alone.
